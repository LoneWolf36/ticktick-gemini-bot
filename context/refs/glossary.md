# Terminology Glossary

> **Purpose:** This glossary provides consistent terminology across all 9 specs (001–009) in the ticktick-gemini project. Each entry includes a clear definition, the owning spec, and distinctions from similar terms to prevent ambiguity during implementation and review.
>
> **Maintenance:** Update this file when new specs introduce terminology that overlaps with existing definitions.

---

## Behavioral Pattern

**Definition:** User-provided self-knowledge about tendencies, habits, or preferences. Stored in `user_context.js` and injected into prompts as contextual guidance.

**Owner Spec:** `user_context.js` (not a numbered spec)

**Distinction:** Unlike behavioral signals or weak inferences, behavioral patterns are **explicitly provided by the user**, not detected or inferred by the system. They represent intentional self-knowledge (e.g., "I work best in the morning", "I tend to procrastinate admin tasks").

**Cross-References:**
- Used by Spec 008 (Work Style and Urgent Mode) for mode transitions
- Referenced in user context initialization

---

## Behavioral Signal

**Definition:** A retained, privacy-bounded event record capturing specific user interactions with the bot (e.g., "task postponed", "scope changed", "task completed early"). Signals are raw data points, not interpretations.

**Owner Spec:** Spec 009 (Behavioral Signals)

**Distinction:** Signals are **raw, factual event records**. They differ from weak inferences (which interpret patterns across signals) and behavioral patterns (which are user-provided, not system-captured). Signals have explicit retention limits and privacy boundaries.

**Cross-References:**
- Consumed by Spec 008 for pattern detection
- Subject to privacy retention policies in Spec 009

---

## Briefing Summary

**Definition:** A structured data output representing the daily plan before it is formatted into user-facing message text. Contains task lists, priorities, scheduling notes, and context for the day ahead.

**Owner Spec:** Spec 006 (Briefing/Review System)

**Distinction:** The briefing summary is a **pre-formatted data structure**. It becomes the briefing message only after passing through the formatter pipeline. This separation allows the same summary to be rendered in different tones or formats.

**Cross-References:**
- Generated by the daily planning pipeline in Spec 006
- Consumed by the formatter module before Telegram delivery

---

## Checklist Intent

**Definition:** A set of sub-steps associated with a single parent task. All checklist items belong to one TickTick task and share its metadata (due date, project, priority).

**Owner Spec:** Spec 005 (Checklist & Multi-Task Handling)

**Distinction:** Checklist intents create **one task with sub-items**, not multiple independent tasks. This differs from multi-task intents, which produce separate tasks that can have different projects, priorities, and due dates.

**Cross-References:**
- Contrasted with Multi-Task Intent in Spec 005
- Processed by the normalizer in Spec 001 when checklist detection is triggered

---

## Humane Mode

**Definition:** The default work-style state for the bot, characterized by a considerate, conversational tone. Used when tasks are not time-critical and the user benefits from contextual, supportive communication.

**Owner Spec:** Spec 008 (Work Style and Urgent Mode)

**Distinction:** Humane mode is the **baseline state**, not a response to urgency. It differs from urgent mode in tone (conversational vs. minimal), message length (detailed vs. terse), and interaction style (supportive vs. action-oriented).

**Cross-References:**
- Paired with Urgent Mode in Spec 008
- Mode transitions governed by the Shared Urgency Utility

---

## Intent Action

**Definition:** A structured object produced by the intent extraction layer representing a single unit of user intent. Contains fields: `type`, `title`, `content`, `priority`, `projectHint`, `dueDate`, `repeatHint`, `splitStrategy`, `confidence`.

**Owner Spec:** Spec 001 (Task Operations Pipeline)

**Distinction:** An intent action is **raw LLM output** and has not yet been validated or normalized. It becomes a Normalized Action only after passing through the normalizer's validation, sanitization, and enrichment steps.

**Cross-References:**
- Input to the normalizer in Spec 001
- Distinct from Mutation Intent (Spec 002) which modifies existing tasks

---

## Life Theme

**Definition:** A user-defined consequential life area (e.g., career, health, family, fitness) stored in `user_context.js`. Used to provide contextually relevant framing for task suggestions and prioritization.

**Owner Spec:** Spec 007 (Execution Prioritization Foundations)

**Distinction:** Life themes are **explicitly defined by the user**, not inferred by the system. They represent areas the user has identified as important, which differs from behavioral patterns (tendencies) and behavioral signals (event records).

**Cross-References:**
- Stored in `user_context.js` alongside Behavioral Patterns
- Referenced by Spec 006 for daily planning context

---

## Multi-Task Intent

**Definition:** An intent that produces multiple independent tasks from a single user message. Each task has its own title, and may have different projects, priorities, and due dates.

**Owner Spec:** Spec 001 (Task Operations Pipeline)

**Distinction:** Multi-task intents create **separate, independent tasks**, not sub-items of a single task. This differs from checklist intents, which produce one parent task with sub-steps. The normalizer handles splitting via the `splitStrategy` field.

**Cross-References:**
- Contrasted with Checklist Intent in Spec 005
- Split logic defined in Spec 001 normalizer

---

## Mutation Intent

**Definition:** A structured request to modify an existing task — specifically to update its properties, mark it complete, or delete it.

**Owner Spec:** Spec 002 (Natural-Language Task Mutations)

**Distinction:** Mutation intents **change existing tasks**, whereas intent actions and normalized actions are focused on task creation. Mutation intents require task identification (via title matching, ID, or context) before the mutation can be applied.

**Cross-References:**
- Consumed by the mutation pipeline in Spec 002
- Requires task lookup before execution

---

## Normalized Action

**Definition:** A validated and cleaned Intent Action that is ready for execution. Processing includes: title truncation, content filtering, `repeatHint` → `repeatFlag` conversion, project hint resolution to actual project ID, and confidence threshold validation.

**Owner Spec:** Spec 001 (Task Operations Pipeline)

**Distinction:** A normalized action is **post-validation, pre-execution**. It differs from an intent action in that all fields have been sanitized, defaults applied, and confidence verified. Only normalized actions proceed to TickTick API calls.

**Cross-References:**
- Output of the normalizer in Spec 001
- Input to the ticktick-adapter execution layer

---

## Pipeline Contract

**Definition:** The shared input/output shape that all pipeline modules (intent extraction, normalizer, adapter) must adhere to. Defines the data structures, field types, and error shapes that flow between modules.

**Owner Spec:** Spec 001 (Task Operations Pipeline)

**Distinction:** The pipeline contract is an **interface definition**, not a runtime object. It ensures that each module can be developed, tested, and replaced independently as long as it honors the contract.

**Cross-References:**
- Defines module boundaries in Spec 001
- Referenced by all specs that extend the pipeline

---

## Shared Urgency Utility

**Definition:** A single, shared module that provides task urgency classification via the `classifyUrgency` function. Used by multiple specs to determine whether a task or situation requires urgent-mode responses.

**Owner Spec:** Spec 008 (Work Style and Urgent Mode)

**Distinction:** Despite being owned by Spec 008, the Shared Urgency Utility is **queried by both Spec 006 and Spec 008**. It is the single source of truth for urgency classification, preventing divergent urgency logic across specs.

**Cross-References:**
- Queried by Spec 006 (Briefing/Review) for task prioritization
- Queried by Spec 008 (Work Style and Urgent Mode) for mode transitions

---

## Urgent Mode

**Definition:** A manually-activated work-style state that changes task ordering and AI tone to minimal, action-oriented communication. Persists until explicitly disabled by the user.

**Owner Spec:** Spec 008 (Work Style and Urgent Mode)

**Distinction:** Unlike Humane Mode (the default baseline), Urgent Mode is an explicit user override. It does not auto-enable based on inferred behavior.

**Cross-References:**
- Queried by Spec 006 for briefing reminders
- Uses the Shared Urgency Utility for task classification

---

## Weekly Summary

**Definition:** A structured data output representing the weekly review before it is formatted into user-facing message text. Contains completed tasks, pending items, trends, and reflection prompts for the week.

**Owner Spec:** Spec 006 (Briefing/Review System)

**Distinction:** Like the briefing summary, the weekly summary is a **pre-formatted data structure**. It is rendered into the user-facing weekly message after passing through the formatter, allowing consistent tone adaptation.

**Cross-References:**
- Generated by the weekly review pipeline in Spec 006
- Uses the same formatter as the Briefing Summary

---

## Weak Inference

**Definition:** A system-generated interpretation of behavioral signals that suggests a pattern but is not treated as fact. Weak inferences inform recommendations but never override explicit user state or behavioral patterns.

**Owner Spec:** Spec 008 (Work Style and Urgent Mode)

**Distinction:** Unlike behavioral signals (raw event records) or behavioral patterns (user-provided self-knowledge), weak inferences are probabilistic interpretations that must be explicitly surfaced and can be rejected by the user.

**Cross-References:**
- Contrasted with Behavioral Signal in Spec 009
- Must not silently enable Urgent Mode (Spec 008)
