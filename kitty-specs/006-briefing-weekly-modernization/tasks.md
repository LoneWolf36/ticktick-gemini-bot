# Work Packages: Briefing and Weekly Pipeline Modernization

**Inputs**: Design documents from `kitty-specs/006-briefing-weekly-modernization/`  
**Prerequisites**: `plan.md`, `spec.md`, `research.md`, `data-model.md`, `contracts/summary-surfaces.openapi.yaml`, `quickstart.md`

**Tests**: Required. The feature spec, planning alignment, and research all require embedded regression work to support TDD and preserve manual/scheduler parity.

**Organization**: Fine-grained subtasks (`Txxx`) roll up into work packages (`WPxx`). Each work package is independently deliverable, reviewable, and sized to stay focused enough for single-session implementation and review.

**Prompt Files**: Each work package references a matching prompt file in `kitty-specs/006-briefing-weekly-modernization/tasks/` generated by `/spec-kitty.tasks`. This file is the checklist and dependency map; the prompt files carry the detailed implementation guidance.

---

## Work Package WP01: Foundation - Shared Summary Surface Contracts (Priority: P0)

**Goal**: Establish the shared summary module boundary, schema surface, and contract-first fixtures that every downstream stream depends on.
**Independent Test**: The shared summary modules import cleanly, contract fixtures exist, and regression checks can assert the fixed top-level section contracts before full behavior is implemented.
**Prompt**: `kitty-specs/006-briefing-weekly-modernization/tasks/WP01-foundation-shared-summary-surface-contracts.md`
**Estimated Prompt Size**: ~340 lines

**Requirements Refs**: FR-003, FR-004, FR-004a, FR-004b, FR-004c, FR-008

### Included Subtasks
- [x] T001 Create `services/summary-surfaces/` with `index.js`, `briefing-summary.js`, `weekly-summary.js`, and `summary-formatter.js` export seams
- [x] T002 Define shared summary schemas and section helpers in `services/schemas.js` and `services/summary-surfaces/index.js`
- [x] T003 [P] Create reusable summary input fixtures/builders in `tests/regression.test.js` and `tests/run-regression-tests.mjs`
- [x] T004 [P] Add contract-first regression checks for fixed top-level sections and non-behavioral weekly watchouts
- [x] T005 Freeze shared import names and temporary orchestration seams so downstream work packages can branch without redefining interfaces

### Implementation Notes
- Keep the new module split minimal and function-based.
- Use the section contracts from `kitty-specs/006-briefing-weekly-modernization/spec.md` as the canonical boundary.
- Do not wire `bot/commands.js` or `services/scheduler.js` yet beyond what is needed to expose stable imports.

### Parallel Opportunities
- T003 and T004 can run in parallel once T001 and T002 define the contract surface.

**Dependencies**: None.

### Risks & Mitigations
- Risk: Over-scaffolding the new folder into a generic framework.
- Mitigation: Limit foundation work to the four files named in the plan and avoid new classes or registries.

## Work Package WP02: Daily Structured Summary Core (Priority: P1)

**Goal**: Replace the legacy daily free-form path with a structured briefing summary object that honors ranking policy, sparse-data honesty, and the fixed daily section contract.
**Independent Test**: Given representative active-task sets, the daily summary builder returns `focus`, `priorities`, `why_now`, `start_now`, and `notices` before formatting, with honest sparse-data fallbacks.
**Prompt**: `kitty-specs/006-briefing-weekly-modernization/tasks/WP02-daily-structured-summary-core.md`
**Estimated Prompt Size**: ~360 lines

**Requirements Refs**: FR-001, FR-004, FR-004a, FR-004b, FR-005, FR-007, FR-008

### Included Subtasks
- [x] T006 Implement structured daily-generation request and response parsing in `services/gemini.js`
- [x] T007 Build the daily summary composer in `services/summary-surfaces/briefing-summary.js`
- [x] T008 Wire `services/execution-prioritization.js` ranking output into daily `focus`, `priorities`, and `why_now` without local policy drift
- [x] T009 Implement sparse-task and degraded-ranking notices with an actionable `start_now`
- [x] T010 Add daily summary regression coverage that inspects structured output before formatting

### Implementation Notes
- Daily summary logic should depend on the shared contracts from WP01 rather than inventing new fields.
- Preserve current tone closely; changes should come from deterministic formatting later, not from this core package.
- Keep daily fallback behavior useful but compact when there are too few strong candidates.

### Parallel Opportunities
- Can run in parallel with WP03 and WP04 after WP01 lands.

**Dependencies**: WP01.

### Risks & Mitigations
- Risk: Daily logic starts re-ranking tasks locally.
- Mitigation: Use shared ranking output as the source of truth and keep composer logic focused on section shaping only.

## Work Package WP03: Weekly Structured Summary Core (Priority: P1)

**Goal**: Replace the legacy weekly free-form path with a structured weekly summary object that supports reduced digests, evidence-backed watchouts, and the fixed weekly section contract.
**Independent Test**: Given representative current tasks and processed-task history, the weekly summary builder returns `progress`, `carry_forward`, `next_focus`, `watchouts`, and `notices`, with explicit missing-history handling and no behavioral interpretation.
**Prompt**: `kitty-specs/006-briefing-weekly-modernization/tasks/WP03-weekly-structured-summary-core.md`
**Estimated Prompt Size**: ~390 lines

**Requirements Refs**: FR-002, FR-004, FR-004a, FR-004c, FR-005, FR-005a, FR-005b, FR-007, FR-008

### Included Subtasks
- [x] T011 Implement structured weekly-generation request and response parsing in `services/gemini.js`
- [x] T012 Build the weekly summary composer in `services/summary-surfaces/weekly-summary.js`
- [x] T013 Implement reduced-digest behavior when processed history is sparse or missing
- [x] T014 Enforce evidence-backed watchout filtering and explicit missing-data notices only
- [x] T015 Use current tasks and existing ranking/state inputs for `next_focus` without behavioral interpretation
- [x] T016 Add weekly summary regression coverage for fallback and watchout boundaries

### Implementation Notes
- Weekly summary logic must keep `watchouts` factual and within the `006` scope boundary.
- Missing history must change the content and notices, not remove top-level sections.
- Avoid carrying prompt-era sections such as `avoidance`, `needle_mover_ratio`, or `callout` into the new contract.

### Parallel Opportunities
- Can run in parallel with WP02 and WP04 after WP01 lands.

**Dependencies**: WP01.

### Risks & Mitigations
- Risk: Weekly output drifts into `009` behavioral-memory territory.
- Mitigation: Treat the evidence-backed `watchouts` rule as a hard acceptance gate and encode it in tests.

## Work Package WP04: Deterministic Summary Formatter (Priority: P1)

**Goal**: Render the structured briefing and weekly summaries into Telegram-safe output while preserving recognizable headers, tone, and urgent-mode reminder behavior.
**Independent Test**: Formatter tests pass for both daily and weekly summaries, section ordering is stable, and output remains Telegram-safe without relying on free-form final copy from the model.
**Prompt**: `kitty-specs/006-briefing-weekly-modernization/tasks/WP04-deterministic-summary-formatter.md`
**Estimated Prompt Size**: ~330 lines

**Requirements Refs**: FR-001, FR-002, FR-004, FR-007

### Included Subtasks
- [x] T017 Implement the daily section renderer in `services/summary-surfaces/summary-formatter.js`
- [x] T018 Implement the weekly section renderer in `services/summary-surfaces/summary-formatter.js`
- [x] T019 Preserve existing headers and urgent-mode reminder semantics through `bot/utils.js`
- [x] T020 Add Telegram-safety guards and stable section-order rules
- [x] T021 Add formatter-focused regression fixtures and assertions for both summary kinds

### Implementation Notes
- Formatter logic owns rendering only; it must not inspect raw task history or re-derive summary policy.
- Reuse existing Telegram-safe helpers where possible to keep visible tone drift minimal.
- Keep section rendering deterministic so manual and scheduled output can be compared directly.

### Parallel Opportunities
- Can run in parallel with WP02 and WP03 after WP01 lands.

**Dependencies**: WP01.

### Risks & Mitigations
- Risk: Formatter changes become a hidden copy rewrite.
- Mitigation: Preserve existing headers/reminders and make any wording changes explicit in regression fixtures.

## Work Package WP05: Manual Command Integration (Priority: P1)

**Goal**: Move manual `/briefing` and `/weekly` command handlers onto the shared summary surface without regressing auth, quota, or manual command behavior.
**Independent Test**: Manual command handlers call the shared summary surface, log structured output before formatting, and preserve current auth/quota error handling and user-facing tone.
**Prompt**: `kitty-specs/006-briefing-weekly-modernization/tasks/WP05-manual-command-integration.md`
**Estimated Prompt Size**: ~350 lines

**Requirements Refs**: FR-001, FR-002, FR-003, FR-004, FR-006, FR-007

### Included Subtasks
- [x] T022 Replace `/briefing` command execution in `bot/commands.js` to call the shared summary surface
- [x] T023 Replace `/weekly` command execution in `bot/commands.js` to call the shared summary surface
- [x] T024 Preserve existing auth, quota, and user-facing error handling while removing legacy string-only path dependence
- [x] T025 Inspect and log structured summary output plus formatter decisions in manual command paths
- [x] T026 Add manual command regression coverage for shared-surface use and tone-sensitive output invariants

### Implementation Notes
- This package should not rework scheduler behavior.
- Keep command-layer changes thin: load data, call shared surface, deliver Telegram reply.
- If manual command handlers need adapter helpers, keep them local to `bot/commands.js` or the shared summary surface rather than introducing a new service tier.

### Parallel Opportunities
- Can run in parallel with WP06 once WP02, WP03, and WP04 are available.

**Dependencies**: WP02, WP03, WP04.

### Risks & Mitigations
- Risk: Manual commands preserve old behavior in one path and shared behavior in another.
- Mitigation: Replace both `/briefing` and `/weekly` in the same package and add explicit regression coverage for both.

**Goal**: Move scheduled daily and weekly sends onto the shared summary surface while preserving scheduler-specific delivery wrappers and ensuring parity with manual paths.
**Independent Test**: Scheduler jobs call the same shared summary surface as manual commands, preserve existing quota/error handling, and keep the pending-review reminder outside the shared summary contract.
**Prompt**: `kitty-specs/006-briefing-weekly-modernization/tasks/WP06-scheduler-integration-and-delivery-parity.md`
**Estimated Prompt Size**: ~340 lines

**Requirements Refs**: FR-003, FR-006, FR-007

### Included Subtasks
- [x] T027 Replace scheduled daily execution in `services/scheduler.js` to call the shared summary surface
- [x] T028 Replace scheduled weekly execution in `services/scheduler.js` to call the shared summary surface
- [x] T029 Keep the pending-review reminder as a scheduler-only post-format delivery wrapper
- [x] T030 Preserve quota skip, auth safety, and delivery-failure behavior while removing legacy string-only path dependence
- [x] T031 Add scheduler-focused parity coverage for the shared summary surface

### Implementation Notes
- Scheduler wrappers should stay thin and reuse the same summary contracts as manual commands.
- Do not move pending-review reminder logic into the shared summary contract.
- Keep scheduler telemetry consistent with manual command telemetry so parity is auditable later.

### Parallel Opportunities
- Can run in parallel with WP05 once WP02, WP03, and WP04 are available.

**Dependencies**: WP02, WP03, WP04.

### Risks & Mitigations
- Risk: Scheduler output diverges subtly from manual output because of delivery-only wrappers.
- Mitigation: Keep wrappers narrow and add parity assertions around the shared summary stage before delivery.

**Goal**: Finish the feature with shared diagnostics, cross-path parity coverage, quickstart validation, and removal of obsolete legacy daily/weekly code paths.
**Independent Test**: Cross-path regressions pass, logs expose source counts and summary diagnostics, quickstart scenarios are satisfied, and dead legacy daily/weekly summary code is removed.
**Prompt**: `kitty-specs/006-briefing-weekly-modernization/tasks/WP07-observability-and-regression-stabilization.md`
**Estimated Prompt Size**: ~320 lines

**Requirements Refs**: FR-003, FR-006, FR-007

### Included Subtasks
- [x] T032 Normalize summary diagnostics field names across manual and scheduled execution
- [x] T033 Log source counts, degraded reasons, structured summary objects, and formatting decisions before delivery
- [x] T034 Add cross-path parity regressions comparing manual and scheduled behavior for equivalent inputs
- [x] T035 Validate `kitty-specs/006-briefing-weekly-modernization/quickstart.md` scenarios and patch any stale implementation notes or comments
- [x] T036 Remove obsolete legacy daily/weekly prompt-only helpers or dead code once parity coverage passes

### Implementation Notes
- Treat this package as the stabilization gate before review, not as a place to invent new feature behavior.
- Prefer removal of unreachable legacy code only after parity tests exist.
- Use the quickstart as a reviewer-facing acceptance checklist.

### Parallel Opportunities
- Best run after WP05 and WP06 converge; do not start dead-code removal before parity coverage exists.

**Dependencies**: WP05, WP06.

### Risks & Mitigations
- Risk: Logging or cleanup changes hide regressions rather than exposing them.
- Mitigation: Land parity tests before cleanup and keep logging field names stable across both paths.

## Dependency & Execution Summary

- **Sequence**: WP01 -> (WP02, WP03, WP04 in parallel) -> (WP05, WP06 in parallel) -> WP07.
- **Parallelization**: The feature is intentionally split so three core streams can branch after WP01, then two adapter streams can proceed in parallel once the core contracts are in place.
- **MVP Scope**: WP01 through WP06. WP07 is the stabilization and cleanup gate before final review.
