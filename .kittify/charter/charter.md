## Product Identity

The system is an **ADHD-friendly execution assistant** that:

- Silently converts natural-language Telegram messages into clean TickTick tasks.
- Prefers TickTick-native recurrence when intent is clear.
- Keeps daily focus constrained and useful.
- Escalates overdue work in a predictable, non-noisy way.
- Reduces clutter, reduces choice overload, and increases trust.
- Never invents completions, never leaks user context into titles, and never requires unnecessary manual review for straightforward task creation.

## Architecture Principles

### 1. Use AX as the LLM orchestration layer

AX is the preferred runtime layer because it fits the existing Node.js/ESM codebase and supports structured LLM programming, validation, tool usage, and provider flexibility without introducing a second language runtime.

AX converts messy Telegram input into typed intent objects. It does **not** own business rules or final execution decisions.

### 2. Use a direct REST API TickTick adapter

TickTick operations must go through a thin local adapter that wraps the direct REST API for task CRUD, project lookup, subtasks, priorities, reminders, and recurring rules. This relies on the battle-tested infra in `ticktick.js` (OAuth2, retry, cache) while exposing a clean adapter contract to the rest of the application.

The app must **not** scatter direct TickTick API behavior across commands and prompts. All TickTick operations flow through the adapter.

### 3. Keep deterministic logic in-app

Validation, workflows, and state management must remain deterministic in code, outside the LLM. This is **non-negotiable**.

The application owns:

- Title length caps
- Content filtering and suppression
- Repeat-hint normalization
- Due-date expansion
- Allowed actions enforcement
- Safety checks before any task reaches TickTick

This rule exists because the original system proved that prompt/schema hints are not sufficient for enforcement.

### 4. Prefer recurrence hints plus server-side mapping

The model emits a recurrence hint (e.g., `daily`, `weekdays`, `weekly`, `every monday`). The application converts that hint into a valid TickTick `repeatFlag` format before calling the adapter.

This preserves control and avoids relying on the model to generate raw recurrence rules.

### 5. Target architecture flow

```
Telegram message -> AX intent extraction -> Deterministic normalization & validation -> TickTick adapter -> TickTick
```

The LLM handles interpretation. The app handles rules and safety. The adapter handles external task operations.

## Behavioral Rules

### Task titles

- Must be short, verb-led where possible.
- Must be free from dates, priorities, project names, or leaked user-context content.

### Task content

- Must only preserve useful references (URLs, locations, specific instructions).
- Must never contain coaching prose, motivational filler, or analysis-card noise.

### Recurrence

- Preferred when the user clearly expresses repeated intent (daily, weekdays, weekly, every Monday).
- Multi-day splitting used when the user names distinct dates for separate one-off sessions (not true recurrence).

### Response behavior

- Silent by default for clear task-creation requests; return only a terse success response.
- Follow-up questions only when a request is genuinely ambiguous in a way that would create wrong tasks.

## Tribal Knowledge

### Conventions and Lessons Learned

Based on debugging and development experience, the following conventions guide future work.

#### 1. Avoid over-engineering

Features and abstractions should only be introduced when necessary. The system
should evolve incrementally rather than anticipating every possible future requirement.

#### 2. Ensure every line of code adds value

Code should exist only if it solves a real problem. Avoid unnecessary abstractions
and redundant logic.

#### 3. Architecture before implementation

Core architecture should be defined before writing large amounts of logic. Ad-hoc
development followed by restructuring leads to wasted effort.

#### 4. Maintain separation of concerns

Each module should have a clear responsibility. Avoid excessive fragmentation into
too many small files.

#### 5. Maintain a project index for LLM collaboration

Because LLMs have context limits, the system should maintain an index describing
architecture, modules, and responsibilities so LLM tools understand the project state.

#### 6. Keep deterministic logic outside the LLM

LLMs should assist with interpretation and reasoning, but validation, workflows,
and state management must remain deterministic in code.

#### 7. Enforce structured outputs from LLMs

All LLM responses must follow strict schemas (for example AX typed outputs) to avoid parsing failures.

#### 8. Implement guardrails and validation

LLM outputs must always be validated before being used. Schema validation, allowed
actions, and sanity checks should protect the system.

#### 9. Prioritize observability and logging

All LLM interactions should be logged, including prompts, responses, decisions,
and errors. The pipeline should capture: raw user request, AX intent output,
normalized actions, adapter requests, adapter results, and validation failures.

#### 10. Explicitly manage system state

Agent systems should track state such as conversation context, task status, and
workflow progress instead of relying purely on message history.

#### 11. Test with messy real-world inputs

The system should be tested with incomplete, ambiguous, or multi-intent inputs
since real-world usage rarely follows clean formats. Example test inputs:
- "practice DSA every weekday"
- "plan Toast recruiter interview"
- "study system design monday tuesday and wednesday"

#### 12. Do not over-trust external layers

Wrap external dependencies (like MCP servers) behind internal adapters, pin
dependencies, log all requests/results, and keep normalization and safety checks
inside the codebase. The adapter contract must be testable independently.

### Historical Decisions

Key architectural decisions so far:

- **Redis over file-based storage:** Chosen to ensure durability across server restarts.
- **Local user context file:** Stored locally in a gitignored file for the single-user setup.
- **AX over raw Gemini API calls:** Chosen because it supports structured LLM programming, validation, and provider flexibility within the Node.js/ESM stack without adding a separate runtime.
- **Direct REST API TickTick adapter over MCP:** Chosen because the existing `ticktick.js` already has battle-tested OAuth2, retry backoff, and project-move rollback logic that an external MCP server lacks. The spec's intent is a contract boundary, not an MCP protocol boundary.
- **Deterministic normalization over prompt enforcement:** Chosen because the original system proved prompt/schema hints are insufficient for reliable enforcement of title limits, content rules, and recurrence formatting.

**Future architecture may support:**

- Multi-user context storage.
- Configurable user profiles.
- Persistent personalization data.

**Users may eventually define:**

- Their goals.
- Their strengths.
- Their struggles.

Agents can then adapt productivity strategies accordingly.
